# RikkaMQ v0.2 API Redesign Plan

Status: 設計確定・段階移行中
Origin: Oracle による v0.1.3 のフルレビュー (設計破綻4件 + 実装ミス10件) を踏まえた完全再設計。
Scope: API は breaking change 前提。クリーンな v0.2 を作る。

---

## 1. 背景: v0.1.3 のレビュー所見

### 設計破綻 (Critical / High, API breaking 必須)

| # | 重大度 | 箇所 | 内容 |
|---|---|---|---|
| 1 | Critical | `src/mq.rs:18-25`, `src/define/redis/mq.rs:54-55` | `MessageQueue::new` の `id_generator: fn() -> I` が message ID 型と Redis consumer/member 名生成を同型に縛っている。`fn` ポインタなので closure 不可。 |
| 2 | High | `src/mq.rs:17-75` | `MessageQueue` trait に enqueue / worker lifecycle / delayed inspector / failed inspector / retry を全部詰めている。delayed/failed は Redis 実装都合の概念。 |
| 3 | High | `src/mq.rs:38-59`, `src/define/redis.rs:304-331` | `get_*_infos(size, offset)` の offset pagination が HSCAN cursor を offset として誤用。HSCAN cursor は opaque であり row offset ではない。trait 契約自体が壊れている。 |
| 4 | High | `src/mq.rs:29-30`, `src/define/redis/mq.rs:202-235` | `start_workers(&self)` が `()` を返し、JoinHandle も cancel token も shutdown もない。worker leak。 |

### 実装ミス (互換修正可能 / 一部 breaking)

| # | 重大度 | 箇所 | 内容 |
|---|---|---|---|
| 5 | Critical | `src/define/redis/mq.rs:113-154` | `push_failed_info` が失敗しても `mark_done` してしまい、ジョブ消失。 |
| 6 | High | `src/define/redis/mq.rs:295-309` | `retry_failed` の delete-before-enqueue。途中失敗でデータロス。 |
| 7 | Confirmed | `src/define/redis/mq.rs:115` | `max_retry` off-by-one (`>` で max_retry+1 回試行)。 |
| 8 | High | `src/define/redis/mq.rs:203-233` | poison 時の真の無限ループ。`i += 1` が spawn 成功時のみで、`continue` ループが無限化。 |
| 9 | Medium | `src/define/redis.rs:92-96, 165-166` | `XGROUP CREATE` を insert_waiting/pop_pending 毎に呼んでいる。 |
| 10 | High | `src/error.rs:10-15`, `src/define/redis/mq.rs:139,366,375` | `ErrorOperation::Delay(format!("{e:?}"))` で source chain 損失。 |
| 11 | Confirmed | `src/define/redis/mq.rs:34` | `Mutex<Box<fn() -> I>>` 不要 (fn は Copy)。 |
| 12 | Confirmed | `src/handler.rs:5-71` | handler type erasure が過剰。単一 handler なら `Arc<dyn Fn(...) -> BoxFuture<...> + Send + Sync>` で十分。 |
| 13 | Medium | `src/define/redis.rs:34-72` | `From<Value>` 系で context 損失。 |
| 14 | Confirmed-with-nuance | listen loop | `XREAD BLOCK 1000` は executor は塞がないが pool connection を最大1秒占有、小 pool 時に `enqueue()` が starve。 |

### Refuted (誤指摘)

- `pop_pending` が `retry_delay` 前に新規 message を claim するという指摘は誤り。XPENDING IDLE と XCLAIM 両方で同じ idle threshold (`src/define/redis.rs:167-200`)。

## 2. 設計原則 (v0.2)

1. **core trait は backend 中立**: delayed / failed / retry を core trait から外し、`backend/redis` 専用の inspector trait に切り出す。
2. **message ID と consumer ID は別概念**: `consumer_id_generator` を独立した closure として受ける。Uuid を message ID に使うかどうかと、consumer 名を何にするかは無関係。
3. **ID 生成は `Arc<dyn Fn() -> _ + Send + Sync>`**: stateful closure 可能。`Mutex<Box<fn()>>` 廃止。
4. **worker lifecycle は `WorkerSet` で明示管理**: `JoinHandle`, `watch::Sender<bool>` (shutdown), `shutdown().await`, `join().await` を提供。
5. **HSCAN cursor は `Cursor` 型に閉じ込め**: `scan_*` は `ScanPage { items, next_cursor }` を返し、`None` で終了を表す。`get_*_infos(size, offset)` の offset pagination は廃止。
6. **handler は単純な `HandlerFn`**: `pub type HandlerFn<M, T> = Arc<dyn Fn(M, T) -> BoxFuture<'static, HandlerResult> + Send + Sync>`。axum 由来の erasure machinery は削除。
7. **`ErrorOperation` で source chain 保持**: `ErrorOperation::{Delay, Fail}(Box<dyn std::error::Error + Send + Sync + 'static>)`。
8. **worker loop は `QueueStore<I, T>` trait に依存** (test seam): Redis 実装をモックなしで in-memory に差し替えられる。

## 4. コア API シグネチャ

```rust
// mq.rs
pub trait MessageQueue<I, T>: Send + Sync + 'static {
    fn enqueue(
        &self,
        info: QueueInfo<I, T>,
    ) -> impl Future<Output = Result<(), Error>> + Send;
}

pub trait QueueStats {
    fn queued_len(&self) -> impl Future<Output = Result<u64, Error>> + Send;
}

// inspect.rs
#[derive(Debug, Clone)]
pub struct Cursor(/* opaque bytes */);

#[derive(Debug)]
pub struct ScanPage<T> {
    pub items: Vec<T>,
    pub next_cursor: Option<Cursor>,
}

pub trait QueueInspector<I, T> {
    fn delayed_len(&self) -> impl Future<Output = Result<u64, Error>> + Send;
    fn failed_len(&self) -> impl Future<Output = Result<u64, Error>> + Send;
    fn get_delayed(&self, id: &I) -> impl Future<Output = Result<Option<QueueInfo<I, T>>, Error>> + Send;
    fn get_failed(&self, id: &I) -> impl Future<Output = Result<Option<ErroredInfo<I, T>>, Error>> + Send;
    fn scan_delayed(&self, limit: usize, cursor: Option<Cursor>)
        -> impl Future<Output = Result<ScanPage<QueueInfo<I, T>>, Error>> + Send;
    fn scan_failed(&self, limit: usize, cursor: Option<Cursor>)
        -> impl Future<Output = Result<ScanPage<ErroredInfo<I, T>>, Error>> + Send;
}

pub trait FailedRetry<I> {
    fn retry_failed(&self, id: &I) -> impl Future<Output = Result<(), Error>> + Send;
}

// worker.rs
pub struct WorkerSet {
    shutdown_tx: tokio::sync::watch::Sender<bool>,
    handles: Vec<tokio::task::JoinHandle<Result<(), Error>>>,
}

impl WorkerSet {
    pub async fn shutdown(self) -> Result<(), Error> { /* signal + join_all */ }
    pub async fn join(self) -> Result<(), Error> { /* join_all only */ }
}

pub trait WorkerControl<M, I, T>
where
    M: Clone + Send + Sync + 'static,
{
    fn start_workers(
        &self,
        module: M,
        handler: HandlerFn<M, T>,
    ) -> impl Future<Output = Result<WorkerSet, Error>> + Send;
}

// handler.rs
pub type HandlerResult = Result<(), ErrorOperation>;
pub type HandlerFn<M, T> = std::sync::Arc<
    dyn Fn(M, T) -> futures::future::BoxFuture<'static, HandlerResult> + Send + Sync,
>;

pub fn into_handler<M, T, F, Fut>(f: F) -> HandlerFn<M, T>
where
    F: Fn(M, T) -> Fut + Send + Sync + 'static,
    Fut: std::future::Future<Output = HandlerResult> + Send + 'static,
{
    std::sync::Arc::new(move |m, t| Box::pin(f(m, t)))
}
```

### Builder (backend/redis/builder.rs)

```rust
let mq = RedisMessageQueue::<Uuid, QueueData>::builder()
    .pool(redis_pool.clone())
    .name("mq_name")
    .config(config)
    .consumer_id_generator(|| format!("consumer-{}", Uuid::new_v4()))
    .build()?;
```

- `M` は builder からは消えて、`WorkerControl::start_workers` で受ける。
- `consumer_id_generator` は `impl Fn() -> String + Send + Sync + 'static`。

### Worker loop 状態遷移 (run_worker 内)

| handler 結果 | delivered_count | アクション |
|---|---|---|
| `Ok(())` | n/a | `record_failed`/`record_delayed` を介さず `ack`。 `delivered_count > 0` なら `remove_delayed` も実行。 |
| `Err(Delay(e))` | `< max_retry` | `record_delayed(info, e)` → `ack`。 |
| `Err(Delay(e))` | `>= max_retry` | `record_failed(info, e)` → `ack`。 |
| `Err(Fail(e))` | n/a | `record_failed(info, e)` → `ack`。 |
| shutdown signal | n/a | loop break、handle が `Ok(())` で返り `join` 可能。 |

ポイント:
- `record_*` が失敗したら `?` で伝播し、ack はしない (= ジョブ消失防止)。これが現状の P0 バグ#5の修正。
- max_retry 比較は `>=` で `delivered_count == max_retry` を最終試行扱い (off-by-one 修正)。
- poison (handler panic / spawn 失敗) でも loop は break / 次の supervisor 試行に委ねる。supervisor 側は無限ループしない。

## 5. 移行ガイド (旧 → 新)

| 旧 | 新 |
|---|---|
| `rikka_mq::define::redis::*` | `rikka_mq::backend::redis::*` |
| `MessageQueue<M, I, T>` | `MessageQueue<I, T>` (M は `WorkerControl` のみ) |
| `MessageQueue::new(..., id_generator, handler)` | `RedisMessageQueue::builder().pool().name().config().consumer_id_generator().build()?` |
| `id_generator: fn() -> I` | `consumer_id_generator: impl Fn() -> String + Send + Sync + 'static` |
| `start_workers(&self)` (`-> ()`) | `start_workers(&self, module, handler).await -> Result<WorkerSet, Error>` |
| `queue(info)` | `enqueue(info)` |
| `get_*_infos(size: i64, offset: i64)` | `scan_*(limit: usize, cursor: Option<Cursor>) -> ScanPage<_>` |
| `ErrorOperation::Delay(String)` | `ErrorOperation::Delay(Box<dyn Error + Send + Sync + 'static>)` |
| `Error::DatabaseError` | `Error::Pool` / `Error::Backend` |
| `Error::ConversionError` | `Error::Serialize` / `Error::Deserialize` / `Error::Protocol { op, detail }` |
| `Handler<M, T>` + erasure machinery | `HandlerFn<M, T>` + `into_handler` |
| `Mutex<Box<fn() -> I>>` | `Arc<dyn Fn() -> String + Send + Sync>` |
| `MQConfig.worker_count: i32` | `NonZeroUsize` |
| `MQConfig.max_retry: i32` | `u32` |
| `ErroredInfo.stack_trace: String` | `ErroredInfo.error: Box<dyn Error + Send + Sync>` (シリアライズ時は `Display` の文字列化) |
| `destructure::Destructure` | 廃止。明示的 getters + `into_parts()` |

### After 使用例

```rust
let mq = RedisMessageQueue::<Uuid, QueueData>::builder()
    .pool(redis_pool.clone())
    .name("mq_name")
    .config(config)
    .consumer_id_generator(|| format!("consumer-{}", Uuid::new_v4()))
    .build()?;

let handler = into_handler(move |module: Arc<Module>, data: QueueData| async move {
    let _con = module.pool.get().await.map_err(|e| ErrorOperation::Delay(Box::new(e)))?;
    // ... do work
    Ok(())
});

let workers = mq.start_workers(module.clone(), handler).await?;

mq.enqueue(QueueInfo::new(Uuid::new_v4(), data)).await?;

tokio::signal::ctrl_c().await.map_err(|e| Error::Backend(Box::new(e)))?;
workers.shutdown().await?;
```

## 6. オープン論点 (実装時に確定)

1. **M module generic を維持 vs closure capture 完全寄せ**: 維持 (現状 API との連続性 + Arc 不要なケースに対応) で進める想定。実装時に再評価。
2. **shutdown を graceful only vs forceful (abort) も提供**: graceful (`shutdown`) を基本にし、`abort()` をオプション提供。
3. **`MessageQueue` を object-safe にするか**: `impl Future` 戻り値は object-safe でない。dyn 必要ケースのために `BoxFuture` 版 (`MessageQueueDyn`) を別途用意するか検討。とりあえず v0.2.0 では非 object-safe で進める。
4. **`QueueInfo` の ID 生成を caller vs library**: caller 側 (現行と同じ)。
5. **retry を固定 delay vs RetryPolicy enum (Fixed/Exponential)**: v0.2.0 は固定 (現行互換)、v0.2.x で RetryPolicy 追加。
6. **inspector を public trait vs Redis backend struct 分離**: public trait (`QueueInspector`) として export、Redis backend が impl。

## 7. 段階的移行計画 (実装ロードマップ)

### Phase 1: P0 minimal fixes (このセッションで実装)

現行 API を維持したまま、Oracle が P0 と指定した bugfix のみ適用。

1. `max_retry` の `>` を `>=` に修正 ([src/define/redis/mq.rs:115])。
2. `push_failed_info` を `mark_done` より前にし、失敗時は `?` で伝播 ([src/define/redis/mq.rs:113-154])。
3. `retry_failed` を enqueue-before-delete 順序に変更 ([src/define/redis/mq.rs:295-309])。
4. listen worker supervisor の `i += 1` ロジック修正 (spawn 失敗・panic でも index を進める) ([src/define/redis/mq.rs:203-233])。
5. `start_workers` を breaking change で `WorkerSet` 返却に変更し、watch::Sender<bool> による shutdown を追加。
6. `ErrorOperation::{Delay, Fail}` を `String` から `Box<dyn Error + Send + Sync>` に変更し、source chain を保持。

### Phase 2: API surface 再編成 (v0.2.0 release)

7. `MessageQueue` trait を `enqueue` だけに縮小、`QueueInspector` / `FailedRetry` / `WorkerControl` / `QueueStats` を切り出す。
8. `id_generator` を `consumer_id_generator` に rename し、message ID 型と分離。`Mutex<Box<fn>>` 廃止。
9. `get_*_infos(size, offset)` を `scan_*(limit, cursor)` に置き換え。
10. `Handler` erasure machinery を `HandlerFn` + `into_handler` に置き換え。
11. `define::redis` を `backend::redis` に rename、`builder.rs`/`inspect.rs`/`store.rs`/`keys.rs` に分割。
12. `QueueStore` trait を切り出して `run_worker` を関数化 (test seam)。
13. `destructure` 依存削除、明示的 getter + `into_parts()`。

### Phase 3: 改善追加 (v0.2.x)

14. `XGROUP CREATE` を builder/start_workers の1回呼び出しに集約。
15. `XREAD BLOCK 1000` を pool 別 connection / dedicated connection に分離して enqueue starvation 回避。
16. `From<Value>` を `TryFrom<Value>` に変更し context 保持。
17. RetryPolicy enum (Fixed/Exponential) 追加。
18. テスト suite (QueueStore in-memory impl で worker loop の単体テスト)。
